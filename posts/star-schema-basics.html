Star schema basics for a daily production dashboard

A single flat table works fine for ten rows. At the scale of a real production dashboard, it's what quietly breaks your DAX measures three months in. Here's the model that holds up.

Why a flat table breaks down

The tempting first move is to build one big table: date, employee, project, task type, count, quality score, all in one sheet. It works for a demo. It stops working the moment you need a measure like "average daily count per employee, filtered by project, compared to last month" — because every dimension is duplicated on every row, and DAX has no clean way to filter one without dragging the others along inconsistently.

The fix: separate facts from dimensions

A star schema splits your data into one fact table (the numbers that happened — counts, scores, hours) and several dimension tables (the things those numbers describe — who, what project, what date, what task type). The fact table holds foreign keys pointing to each dimension.

Dim_Employee
Dim_Date

Fact_Production
Dim_Project
Dim_TaskType

Each dimension connects to the fact table with a single relationship, always filtering from dimension to fact (one-to-many). This is what lets a slicer on "Project" correctly filter every measure without you writing custom logic for each one.

What goes in the date dimension

This is the one people skip and regret. A dedicated date table — not just the dates that happen to appear in your fact table — is what makes rolling averages, month-over-month comparisons, and "same day last month" measures possible at all in DAX. It needs one row per calendar day, plus columns for week number, month name, and fiscal period if your reporting cycle isn't a calendar month.

A measure that only works with this structure

Once the model is in place, a rolling 7-day average measure becomes a few lines instead of a workaround:

  • CALCULATE(AVERAGE(Fact_Production[Count]), DATESINPERIOD(Dim_Date[Date], MAX(Dim_Date[Date]), -7, DAY))

Try writing that against a flat table with repeated date columns and you'll end up filtering on the wrong grain half the time — usually silently, which is worse than an error.

Note: Figures and table names here are illustrative. The modeling principle is what to take away — apply it to your own team's actual dimensions.

The one-time cost that pays off

Restructuring an existing flat-table report into a star schema takes a focused afternoon, not a rebuild. The payoff shows up every time someone asks for "one more view" — instead of a new flat export, it's a new measure against a model that already knows how everything relates.
